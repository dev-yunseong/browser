# 아우라 브라우저 코어 아키텍처 근본적 결함 분석 보고서 (Commit: 96d68ad 기준)

사용자님의 지적대로, 현재 브라우저의 레이아웃 및 렌더링 코어 로직은 근본적으로 파탄(Broken) 나 있는 상태입니다. 겉보기 버그(검은 블록, 왼쪽 쏠림)를 고치기 위해 단편적인 로직을 덧붙이는 행위는 전체 시스템을 더 망가뜨릴 뿐이었습니다(버튼 클릭 먹통 등).

전체 코드를 스터디하고 분석한 결과, 엔진 전반에 걸쳐 **4가지 거대한 설계 결함**이 존재함을 확인했습니다.

## 1. 이벤트 핸들링과 레이아웃의 결합도 문제 (버튼이 안 눌린 이유)
*   **문제점**: `main.rs`에서 마우스 클릭 이벤트(`hit_test`)를 처리할 때, 레이아웃 엔진(`src/layout.rs`)이 만들어낸 `LayoutBox` 트리의 절대 좌표(`Rect`)에 전적으로 의존하고 있습니다.
*   **치명적 결함**: `build_layout_tree` 함수가 `onclick` 같은 속성을 읽어서 `event_handlers` 맵에 저장해야 하는데, 레이아웃 계산 로직을 조금이라도 수정하거나 요소를 감싸는 로직이 변경되면 이벤트 데이터가 누락되거나 좌표가 틀어집니다. 
*   **결과**: 제가 이전 작업에서 레이아웃을 고친다며 `LayoutBox` 생성 부분을 건드렸을 때, 이벤트 수집 로직이 누락/파괴되어 버튼 클릭이 먹통이 된 것입니다.

## 2. 단일 패스(Single-pass) 레이아웃 엔진의 한계 (요소 겹침, 쏠림의 이유)
*   **문제점**: 보통의 브라우저는 1. DOM 트리 생성 -> 2. 렌더 트리 생성 -> 3. 크기 계산(Measure) -> 4. 위치 배치(Position)의 다단계(Multi-pass)를 거칩니다. 하지만 현재 코드는 `build_layout_tree`라는 거대한 함수 하나에서 스타일 파싱, 크기 계산, 줄바꿈, x/y 절대 좌표 배치를 **동시에** 처리하고 있습니다.
*   **치명적 결함**: 
    1.  **가짜 텍스트 측정**: 텍스트의 길이를 실제 폰트 크기로 재지 않고 `8.0 * (font_size / 16.0)`이라는 주먹구구식 공식으로 추측합니다. 실제 글자보다 박스가 작게 계산되어 무조건 겹칩니다.
    2.  **박스 모델 붕괴**: `padding`과 `margin`이 요소의 `width`에 어떻게 영향을 미치는지에 대한 명확한 규칙(`box-sizing`)이 없고, `current_x`, `current_y`를 자식에게 넘겨가며 상태를 더럽히고 있습니다.
    3.  **Flexbox 억지 구현**: Flex 레이아웃에서 자식 요소들의 좌표를 강제로 옮기는 `offset_layout_box` 함수를 덧붙여놔서, 마진 자동 계산(`margin: auto`)이나 인라인 요소의 줄바꿈 로직과 심각하게 충돌합니다.

## 3. 수동 픽셀 렌더링의 위험성 (검은 블록의 이유)
*   **문제점**: `src/render.rs`의 텍스트 렌더링(`render_text_wrapped`)에서 `tiny-skia`의 안전한 그리기 API를 쓰지 않고, 픽셀 배열(`mask.data_mut()` 또는 `pixmap.data_mut()`)에 직접 접근하여 루프를 돌며 알파(투명도) 값을 곱하고 있습니다.
*   **치명적 결함**: 안티앨리어싱이나 알파 블렌딩(미리 곱해진 알파, Premultiplied Alpha) 수학 계산이 조금만 어긋나도 해당 픽셀이 완전히 불투명한 검은색 찌꺼기(Artifact)로 남게 됩니다. 박스 섀도우(`render_box_shadow`) 역시 반투명한 사각형을 여러 겹 덧칠하는 비효율적이고 오류가 나기 쉬운 방식을 씁니다.

## 4. JS-Rust 브릿지 상태 불안정
*   **문제점**: JavaScript에서 DOM을 변경할 때(`__aura_style_log` 등), Rust 엔진은 무식하게 전체 DOM을 재탐색하고 거대한 레이아웃 트리를 처음부터 다시 만듭니다. 이 과정에서 포커스 상태나 이벤트 리스너의 상태가 초기화되어 상호작용이 뚝뚝 끊깁니다.

---

## 결론 및 재설계(Re-architecture) 방향
"이 로직이 병신이다"라는 지적이 정확했습니다. 현재의 `build_layout_tree` 함수에 `if` 문을 추가해서 버그를 잡으려는 시도는 모래성에 기둥을 세우는 격입니다.

다음과 같이 근본적인 **엔진 재설계(Re-architecture)**가 필요합니다.

1.  **레이아웃 파이프라인 분리**: `build_layout_tree`를 버리고, `Box Tree 생성` -> `Compute Sizes (실제 폰트 측정 포함)` -> `Compute Positions (x, y 좌표 할당)` 3단계로 명확히 나눕니다.
2.  **라인 박스(Line Box) 도입**: 인라인 요소들이 화면 밖으로 나갈 때 단순히 `x`를 0으로 되돌리는 짓을 멈추고, 한 줄에 들어갈 요소들을 모아 높이를 정렬하는 표준 텍스트 레이아웃 알고리즘을 도입해야 합니다.
3.  **렌더링 추상화**: 픽셀을 직접 건드리는 로직을 전면 삭제하고, `tiny-skia`의 `Path`와 `Fill` 명령어로만 화면을 그리도록 렌더러를 100% 교체해야 합니다.
